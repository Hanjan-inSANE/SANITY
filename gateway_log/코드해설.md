# 코드 해설 — gateway_log (6번 Gateway · 7번 Log 전부 뜯어보기)

이 문서는 `gateway_log/` 안의 **모든 파일을 초심자도 이해되게** 풀어쓴 것이다.
지어낸 설명이 아니라 실제 코드가 하는 일을 그대로 옮겼다.

---

## 0. 큰 그림 — 데이터가 어떻게 흐르나

우리가 만든 건 두 덩어리다.

- **`gateway/`** — "창구"를 켜는 **설정과 실행 파일들**. 여기엔 우리가 짠 파이썬 코드가 없다.
  이미 만들어진 프로그램(LiteLLM)을 **설정해서 도커로 켜는 것**이 전부다.
- **`src/`** — 우리가 짠 **파이썬 코드 두 묶음**.
  - `sanity_llm/` : 에이전트가 창구를 통해 AI를 부르는 도구 (6번)
  - `sanity_log/` : 일어난 일을 파일에 적는 도구 (7번)

한 번의 AI 호출이 흐르는 길:

```
에이전트가 gw.complete("sane-sonnet", "질문") 호출
      │
      ▼
sanity_llm/client.py  ──(가상키로 요청)──►  gateway/ 의 LiteLLM 창구  ──►  진짜 Claude
      │                                                                       │
      │◄──────────────────── 결과(글 + 토큰수 + 비용) ◄─────────────────────────┘
      │
      ▼
sanity_log/writer.py  ──►  logs/comp3.xxxx.jsonl  (한 줄 기록)
```

핵심 원칙 두 개만 기억하면 된다.
1. 에이전트는 **진짜 키를 모른다.** 창구 주소와 한도 걸린 가상키만 안다.
2. AI를 부를 때마다 **자동으로 기록**이 남는다.

## 1. `gateway/` — 창구를 켜는 설정 (코드 아님)

### 1-1. `config.yaml` — 창구의 규칙표

창구(LiteLLM)에게 "어떤 AI를 쓸지, 실패하면 어떻게 할지"를 알려주는 표다.

**`model_list` (모델 메뉴판): 별칭 → 진짜 모델**
우리 에이전트는 `sane-sonnet` 같은 **별칭**만 부르고, 창구가 이걸 진짜 모델로 바꿔준다.

| 별칭 | 진짜 모델 | 언제 쓰나 |
|---|---|---|
| `sane-sonnet` | anthropic/claude-sonnet-4-6 | 기본 (대부분의 작업) |
| `sane-opus` | anthropic/claude-opus-4-8 | 어려운 추론 |
| `sane-haiku` | anthropic/claude-haiku-4-5 | 가볍고 싼 작업 |
| `sane-ground` | anthropic/claude-sonnet-4-6 | 웹검색이 필요한 지식조사 전용 (폴백 금지) |
| `sane-oai-fallback` | openai/gpt-4.1 | Claude가 다 죽었을 때 대타 |
| `sane-local` | ollama/qwen2.5:72b | 인터넷 없이 내 컴퓨터 무료 모델 |

> ⚠ 별칭 뒤의 진짜 모델 ID(`claude-sonnet-4-6` 등)는 threat_modeler가 쓰던 값에 맞춘 **자리표시자**다.
> LiteLLM이 이 이름을 실제로 알아듣는지는 창구를 켜봐야 확정된다(그래서 config에 경고 주석을 달아둠).

**`litellm_settings` (창구 동작 규칙)**
- `num_retries: 2` — 전송이 실패하면 창구가 스스로 2번까지 다시 시도.
- `request_timeout: 600` — 600초 기다려도 답 없으면 포기.
- `drop_params: false` — "AI가 못 알아듣는 옵션은 조용히 버려" 기능을 **끔**. (켜두면 웹검색·캐시 같은 우리가 꼭 필요한 기능이 소리 없이 사라질 수 있어서 껐다.)

**`general_settings` (관리 설정)**
- `master_key` — 창구의 **관리자 마스터키**. 이걸로 선불카드(가상키)를 발급한다. 예산 담당자만 씀.
- `database_url` — 발급한 카드와 쓴 금액을 저장할 **Postgres 주소**.

### 1-2. `docker-compose.yml` — 창구를 실제로 켜는 파일

`docker compose up` 하면 컨테이너 두 개가 뜬다.

- **`litellm`** : 창구 본체.
  - `image` : 이미 만들어진 LiteLLM 공식 이미지를 가져다 씀 (우리가 안 만듦).
  - `command` : "이 설정파일로, 4000번 포트로 켜라".
  - `environment` : **진짜 프로바이더 키가 여기에만 들어간다.** (에이전트 쪽엔 절대 안 넣음)
  - `healthcheck` : 창구가 살아있는지 파이썬으로 확인. (curl이 이미지에 없을 수 있어서 파이썬으로 함)
- **`litellm-db`** : Postgres 데이터베이스.
  - **왜 필요?** 선불카드별로 "얼마 남았나"를 어딘가 저장해야 하니까. 재시작해도 안 사라지게.

### 1-3. `.env.example` — 값 채우는 템플릿

이걸 복사해서 `.env`로 만들고 진짜 값을 채운다. 두 종류가 섞여 있다.
- **창구용** : `SANITY_LITELLM_MASTER_KEY`, `ANTHROPIC_API_KEY` 등 (진짜 키)
- **에이전트용** : `LITELLM_API_BASE`(창구 주소), `LITELLM_API_KEY`(가상키), `SANITY_LOG_DIR`(로그 폴더)

> `.env`는 진짜 키가 들어가므로 **git에 올리면 안 된다.**

## 2. `src/sanity_log/` — 7번 "일지" (제일 쉬운 것부터)

### 2-1. `schema.py` — 기록 "한 줄"의 모양

로그 한 줄이 어떻게 생겼는지 정의한다. 두 가지 틀이 있다.

**`Event` — 모든 사건의 공통 봉투**
"무슨 일이, 언제, 어디서 났나"를 담는 기본 틀.

| 필드 | 뜻 |
|---|---|
| `component` | 몇 번 컴포넌트에서 났나 ("3"=공격자) |
| `event_type` | 사건 종류 (`llm_call` / `status` / `tool_call` / `artifact` / `error`) |
| `trace_id` | 작업 추적번호 (같은 작업의 기록을 나중에 이 번호로 묶음) |
| `scope_id` | 더 세부 단위 (예: `tree3.node7.attacker`) |
| `state` | 상태 (`PENDING`/`RUNNING`/`SUCCESS`/`FAIL`) |
| `payload_ref` | 큰 데이터는 여기에 "위치"만 (`artifact://...`) |
| `ts` | 발생 시각 (자동으로 지금 시간) |
| `extra` | 그 밖의 자유 필드 |

`to_dict()`가 하는 일: 이 객체를 딕셔너리로 바꾸면서 ① `extra` 안의 값들을 바깥으로 펼치고
② 비어있는(None) 필드는 버린다. → 깔끔한 한 줄이 됨.

**`LLMCallRecord` — AI를 한 번 부른 기록 (전용 틀)**
공통 봉투에 더해 AI 호출에만 있는 정보(토큰·비용 등)를 담는다.

| 필드 | 뜻 |
|---|---|
| `model` / `provider` | 어떤 모델/회사가 답했나 |
| `attempt` | 몇 번째 시도인가 |
| `prompt_tokens` / `completion_tokens` | 입력/출력 토큰 수 |
| `cache_read_tokens` / `cache_creation_tokens` | 캐시로 아낀/만든 토큰 (관측용) |
| `cost_usd` | 이번 호출 비용 (LiteLLM이 계산해준 값을 그대로 씀) |
| `latency_ms` | 몇 밀리초 걸렸나 |
| `stop_reason` | 왜 끝났나 (`end_turn` 등) / 실패면 실패종류 |

`to_event()`가 하는 일: 이 AI 기록을 위의 공통 봉투(`Event`)로 감싼다.
→ 그러면 일반 사건이든 AI 호출이든 **똑같이 한 줄로 저장**할 수 있다.

### 2-2. `mask.py` — 비밀 가리기

로그에 비밀번호·키가 그대로 남으면 안 된다. 적기 전에 가린다. 정규식 세 개를 쓴다.

- `_KV` : `api_key=...`, `token: ...`, `password ...` 같은 "이름=값" 형태에서 **값을 가림**.
- `_SK` : `sk-` 로 시작하는 흔한 키 형태를 직접 가림.
- `_BEARER` : `Bearer <토큰>` 형태의 토큰을 가림.

**순서가 중요하다.** `_BEARER`를 **먼저** 돌려야 한다.
안 그러면 `_KV`가 "Authorization: **Bearer**"에서 "Bearer"만 값으로 먹고 정작 진짜 토큰을 남긴다.
(이건 실제로 테스트가 잡아낸 버그라 순서를 바로잡았다.)

- `redact_text(문자열)` : 한 문자열을 가림.
- `redact_dict(딕셔너리)` : 딕셔너리 안의 모든 문자열 값을 재귀적으로 가림. (writer가 기록 직전에 이걸 부름)

### 2-3. `writer.py` — 실제로 파일에 적는 도구 ★

`LogWriter`가 심장이다.

**`__init__` (만들 때):**
- 로그 폴더를 정하고(`SANITY_LOG_DIR` 또는 `./logs`), 없으면 만든다.
- **파일 이름에 프로세스 번호+랜덤을 붙인다** → `comp3.12345-ab12cd34.jsonl`.
  - 왜? 여러 일꾼이 동시에 돌 때 **각자 자기 파일**에 쓰게 하려고. 한 파일에 여럿이 동시에 쓰면 글씨가 뭉개진다(특히 윈도우). 나중에 추적번호로 합치면 된다.
- 파일을 "이어쓰기(append)" 모드로 열어둔다.

**`write(event)` (한 줄 적기):**
1. `redact_dict`로 비밀을 가리고,
2. JSON 한 줄로 만들고 (`ensure_ascii=False` 라 한글이 안 깨짐),
3. 잠금(lock)을 걸고 한 줄 쓴 뒤 **바로 flush** 한다.
   - flush = "지금 당장 디스크로 내보내라". 프로세스가 갑자기 죽어도 그전 기록은 살아남게.

**편의 함수들:**
- `log.event("status", trace_id="t1", state="RUNNING")` — 일반 사건 기록.
- `log.llm_call(record)` — AI 호출 기록 (record를 봉투로 감싸 저장).
- `log.close()` — 마무리. `with LogWriter(...) as log:` 로 쓰면 자동으로 닫힌다.

### 2-4. `__init__.py` — 밖에서 뭘 꺼내 쓰나

`from sanity_log import LogWriter, LLMCallRecord` 처럼 바로 쓸 수 있게 정리해둔 것.

## 3. `src/sanity_llm/` — 6번 "창구 클라이언트"

### 3-1. `errors.py` — 실패 종류 구분

AI 호출이 실패했을 때 "어떤 실패냐"에 따라 대응이 다르다.

| 종류 | 뜻 | 대응 |
|---|---|---|
| `RATE_LIMIT` | 너무 빨리/많이 불렀다 (429) | 다른 모델로 폴백 시도 |
| `TRANSIENT` | 일시적 오류(타임아웃 등) | 폴백 시도 |
| `CONTEXT` | 입력이 너무 길다 | 폴백 안 함 (입력을 줄여야 함) |
| `CONTENT` | 콘텐츠 필터에 걸림 | 폴백 안 함 (상위 보고) |
| `AUTH` | 키 문제 | 폴백 무의미 (즉시 실패) |
| `OTHER` | 그 외 | 즉시 중단 |

- `classify(예외)` : 에러의 종류/메시지를 보고 위 중 하나로 분류.
- `FALLBACKABLE = {TRANSIENT, RATE_LIMIT}` : **폴백해도 되는 종류** 목록.
  - `RATE_LIMIT`을 포함한 이유: 창구가 이미 재시도를 다 하고도 429면, 그 모델이 포화됐다는 뜻 → 다른 모델로 넘기는 게 맞다.

### 3-2. `budget.py` — 선불카드(가상키) 발급 (2.2 대행)

원래 카드 발급은 "2.2 Allocator"가 하는데 아직 담당이 없다. 그래서 임시로 여기서 대신한다.

`issue_key(...)` 가 하는 일: **마스터키로** 창구의 `/key/generate`를 호출해 가상키를 하나 만든다.
- `key_alias` : 이 카드의 이름 (예: `tree3.attacker`).
- `max_budget` : 카드 한도 (달러).
- `models` : 이 카드로 쓸 수 있는 모델 목록.
- `rpm_limit`/`tpm_limit`/`budget_duration`/`metadata` : 속도 한도·기간·부가정보 (있으면 넣음).
- 성공하면 카드 문자열(`sk-...`)을 돌려준다.

> 2.2 담당이 정해지면 이 함수를 그쪽에서 부르기만 하면 된다.

### 3-3. `client.py` — 심장. 에이전트가 AI를 부르는 도구 ★★

`GatewayClient`가 전부다.

**`__init__` (만들 때):**
- 창구 주소(`LITELLM_API_BASE`)와 가상키(`LITELLM_API_KEY`)로 `openai` 표준 클라이언트를 준비한다.
  (OpenAI 회사 AI를 쓰는 게 아니라, "OpenAI와 같은 방식"으로 우리 창구에 말을 거는 것뿐이다.)
- 로그 라이터(`log`)를 받아둔다(없으면 로그 안 남김).

**`complete(...)` — AI를 한 번 부르는 핵심 함수.** 순서대로:
1. **메시지 만들기** — `system`(역할 지시)과 `user`(질문)로 대화를 만든다. (이미 만든 대화 `messages`를 주면 그걸 그대로 씀 — 자가수리에 쓰임)
2. **시도 순서 정하기** — `chain = [기본모델] + [폴백들]`. 단 `sane-ground`(웹검색용)는 폴백 금지.
3. **창구에 요청** — 목록의 모델을 하나씩 시도한다.
   - 요청에 `metadata`로 `trace_id`·`scope_id`를 실어 보낸다 → 로그를 묶는 데 씀.
   - `with_raw_response`로 부르는 이유: 응답 **헤더**에서 LiteLLM이 넣어준 **비용**을 읽으려고.
4. **성공하면** → 결과를 정리(`_to_result`)하고 **로그에 SUCCESS 기록** 후 돌려준다.
   **실패하면** → 실패 종류를 분류하고 **로그에 FAIL 기록**. 폴백 가능한 실패면 다음 모델로, 아니면 즉시 중단.
5. 다 실패하면 에러를 낸다.

**`complete_json(...)` — JSON을 받아야 할 때 (위협모델러 등).**
- 먼저 `complete`로 답을 받고 JSON으로 해석해본다.
- 깨진 JSON이면 **1번 자가수리**: 이때 **원래 질문 + 방금 받은 나쁜 답 + "다시 JSON만 줘"** 를
  대화로 이어붙여 재요청한다.
  - 이게 중요한 이유: 그냥 "JSON 줘"만 보내면 AI는 **뭘** JSON으로 만들지 모른다. 원래 맥락을 같이 줘야 제대로 고쳐준다. (이 부분이 처음엔 버그였는데 고쳤다.)

**속을 채우는 도우미들:**
- `_to_result(...)` : 창구 응답에서 글·토큰수·**캐시 토큰**·**비용(헤더에서)**·모델명을 뽑아 정리.
  - 비용은 **우리가 계산하지 않고** LiteLLM이 준 값을 그대로 쓴다(이중계산 방지).
- `_log(...)` / `_log_fail(...)` : 위 정보로 `LLMCallRecord`를 만들어 로그에 남긴다.
- `_infer_provider(모델명)` : 모델명을 보고 회사를 추정 (`claude`→anthropic 등). 빈칸 방지용.
- `_extract_json(글)` : 글에서 JSON만 뽑아낸다. 코드블록(```)·뒤따라오는 쉼표 같은 흔한 실수를 자동으로 정리.

### 3-4. `__init__.py`

`from sanity_llm import GatewayClient` 로 바로 쓰게 정리한 것.

## 4. `tests/` — "진짜 되는지" 증명하는 코드

테스트는 프로그램이 맞게 동작하는지 **자동으로 확인**하는 코드다. AI나 도커 없이도 돌아간다.

### 4-1. `test_log.py`
- 마스킹이 비밀을 가리는지, 로그 2줄이 파일에 제대로 쌓이는지 확인.

### 4-2. `test_client_mock.py`
- **목(mock)** = "가짜". 진짜 창구 대신 **가짜 창구**를 끼워서, 네트워크 없이 클라이언트 로직만 시험한다.
- 확인 항목:
  - **B1**: 자가수리가 원래 질문·나쁜 답을 포함해 다시 묻는가 → ✅
  - **B2**: 캐시 토큰이 결과·로그에 잡히는가 → ✅
  - **B7**: provider가 빈칸이 아닌가 → ✅
  - **B8**: `Bearer <토큰>`이 가려지는가 → ✅
- 이 테스트를 돌려 실제로 전부 통과하는 걸 확인했다.

## 5. 실제 한 번의 호출을 코드로 따라가기

에이전트(예: 공격자)가 쓰는 코드는 이게 전부다:

```python
from sanity_llm import GatewayClient
from sanity_log import LogWriter

log = LogWriter(component="3")                       # 내 전용 일지 열기
gw = GatewayClient(component="3", log=log)           # 창구 도구 준비(가상키는 환경변수에서)

res = gw.complete(
    "sane-sonnet",                                   # 별칭으로 모델 지정
    user="이 노드의 취약점을 분석해줘...",
    trace_id="tree3-path2",                          # 이 작업의 추적번호
    scope_id="tree3.node7.attacker",                 # 세부 단위
)
print(res.text, "| 비용:", res.cost_usd)
log.close()
```

이 몇 줄 뒤에서 실제로 일어나는 일:
1. `gw.complete`가 대화를 만들고, 가상키로 창구(LiteLLM)에 요청한다.
2. 창구가 예산을 확인하고(한도 넘으면 거부), 진짜 Claude로 보낸다.
3. Claude 답이 오면 토큰·비용을 뽑아 `res`로 돌려주고,
4. 동시에 `log`가 `logs/comp3.xxxx.jsonl`에 그 호출을 한 줄 기록한다.
